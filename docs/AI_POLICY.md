# PuzzlePod AI Code Assistant Policy

**Version:** 1.0
**Last Updated:** 2026-03-25
**Maintainer:** Francis Chow
**Repository:** [https://github.com/LobsterTrap/PuzzlePod](https://github.com/LobsterTrap/PuzzlePod)

---

## 1. Purpose

This policy governs the use of AI code assistants (Claude Code, GitHub Copilot, and
similar tools) in PuzzlePod development. PuzzlePod is a security-critical governance
daemon for AI agent workloads on Linux. The code it produces directly controls
containment boundaries -- Landlock rulesets, seccomp filters, namespace isolation,
cgroup limits, and OPA/Rego policy evaluation. Errors in this code can result in
container escapes or uncontained agent execution.

AI code assistants accelerate development when used responsibly. This policy ensures
that AI-assisted contributions maintain the same quality, security, and auditability
standards as fully human-authored code.

---

## 2. Attribution Rules

All commits that include AI-assisted code must include a commit trailer indicating
the level of AI involvement. Use one of the two trailers below.

### 2.1 Assisted-by (Default)

Use `Assisted-by` when a human directed the work, wrote the prompt, reviewed the
output, and edited the result meaningfully. This is the most common case.

```
Assisted-by: Claude Code <noreply@anthropic.com>
```

Examples of `Assisted-by` usage:
- You asked the AI to generate a test skeleton and then wrote the assertions yourself
- You used the AI for boilerplate (struct definitions, clap derive macros) and wrote
  the logic
- You asked the AI to refactor existing code and reviewed every change

### 2.2 Generated-by

Use `Generated-by` when a large block of code was generated by the AI with minimal
human editing. This signals to reviewers that extra scrutiny is warranted.

```
Generated-by: Claude Code <noreply@anthropic.com>
```

Examples of `Generated-by` usage:
- An entire test file was generated from a prompt describing the test plan
- A serialization/deserialization implementation was generated and used as-is
- A Rego policy was generated from a natural language specification

### 2.3 Trailer Placement

Attribution trailers go after the commit message body and the `Signed-off-by` line:

```
feat(puzzled): add Landlock network ruleset support

Implement Landlock ABI v5 network access control for TCP bind and
connect operations. Profiles can now specify allowed ports.

Signed-off-by: Developer Name <developer@example.com>
Assisted-by: Claude Code <noreply@anthropic.com>
```

## 3. Data Handling

### 3.1 Never Include in Prompts

The following must never be included in AI tool prompts, context windows, or
conversations:

- **Secrets and credentials**: API keys, tokens, passwords, private keys, certificates
- **Personally identifiable information (PII)**: names, email addresses, IP addresses
  of real users
- **Production configuration**: real hostnames, database connection strings, internal
  network topology
- **Proprietary third-party code**: code from dependencies or vendors that is not
  open source, unless license terms explicitly permit it

### 3.2 AI Tool Credential Access

AI code assistants must not have access to:

- Production credentials or secrets stores
- Production D-Bus sessions or `puzzled` daemon instances
- Real agent workload data or governance audit logs
- SSH keys, GPG keys, or signing certificates

### 3.3 License Review

Before using AI-generated code that resembles or references third-party
implementations:

1. Verify the source license is compatible with PuzzlePod's license (MIT)
2. If the AI output closely matches a GPL, LGPL, or AGPL licensed work, do not use
   it without legal review
3. Run `cargo deny check licenses` as part of CI to catch dependency license issues

## 4. Human Accountability

### 4.1 Human Review is Mandatory

- A human developer reviews and approves all AI-generated or AI-assisted code before
  it is merged
- The human reviewer is accountable for the correctness, security, and quality of the
  merged code -- not the AI tool
- AI code review comments (from automated review bots) are advisory only and do not
  count toward required approvals

### 4.2 Reviewer Responsibilities

When reviewing AI-assisted code, the human reviewer must:

1. Verify the code does what the commit message claims
2. Check for hallucinated APIs, nonexistent functions, or incorrect type signatures
3. Confirm that error handling follows PuzzlePod conventions (fail closed, `anyhow`
   for application errors, `thiserror` for library errors)
4. Run the code locally or verify CI passes
5. Check that `unsafe` blocks have a `// SAFETY:` comment explaining the invariant

## 5. Review Requirements

### 5.1 Standard Changes

All pull requests require **1 human approval** before merge. This applies to:

- Feature implementation in `crates/puzzlectl/`, `crates/puzzled-types/`
- Documentation updates in `docs/`
- CI/CD changes in `.github/workflows/`
- Test additions in `crates/*/tests/`
- Profile definitions in `policies/profiles/`

### 5.2 Security-Sensitive Changes

The following paths require **2 human approvals** before merge:

| Path | Reason |
|------|--------|
| `crates/puzzled/src/sandbox/` | Kernel enforcement boundary (Landlock, seccomp, namespaces, cgroups) |
| `crates/puzzled/src/sandbox/seccomp/` | Syscall filter definitions |
| `policies/rules/` | OPA/Rego governance policies (commit/reject decisions) |
| `selinux/` | SELinux type enforcement for `puzzlepod_t` domain |
| `bpf/` | BPF LSM programs (kernel-space enforcement) |
| `crates/puzzled/src/dbus.rs` | D-Bus API surface (privilege boundary) |
| `crates/puzzle-init/` | Container init process (PID 1 in sandbox) |

### 5.3 CODEOWNERS

Configure `.github/CODEOWNERS` to enforce the 2-approval requirement automatically
for security-sensitive paths.

## 6. Prohibited Uses

### 6.1 Autonomous Governance Decisions

AI code assistants must not be used to make autonomous governance decisions in
production PuzzlePod deployments. PuzzlePod enforces deterministic OPA/Rego policy
evaluation by design (see PRD Section 4, Design Principle 5). No ML, heuristics,
or probabilistic decision-making is permitted in the governance path.

### 6.2 Cryptographic Implementations

AI must not generate custom cryptographic primitives, hash functions, key derivation
functions, or encryption routines. Use established crates (`ring`, `rustls`,
`sha2`, `hmac`) from the Rust ecosystem. AI may generate code that *calls*
cryptographic APIs but must not implement the algorithms themselves.

### 6.3 Security Control Circumvention

AI must not be used to:

- Disable or weaken seccomp filters, Landlock rulesets, or namespace isolation
- Generate code that bypasses OPA/Rego policy evaluation
- Create backdoors, privilege escalation paths, or policy exceptions without
  explicit human approval and documentation
- Modify SELinux policy to grant `unconfined_t` or equivalent broad permissions

### 6.4 Unsafe Code Generation

AI-generated `unsafe` Rust code requires explicit justification in the PR
description. The reviewer must verify:

- The `unsafe` block is necessary (no safe alternative exists)
- The safety invariant is documented in a `// SAFETY:` comment
- The block is as small as possible

## 7. Proactive Warnings

AI code assistants used in PuzzlePod development should be configured to issue
proactive warnings in the following situations:

### 7.1 Secrets and PII Detection

If a user pastes content that appears to contain secrets (API keys, tokens,
passwords, connection strings) or PII (email addresses, real names associated with
data), the AI should:

1. Warn the user immediately
2. Recommend removing the sensitive content
3. Decline to include the sensitive content in any generated output

### 7.2 Upstream Policy Verification

When contributing AI-assisted code to upstream projects or accepting contributions
from external contributors:

1. Verify the upstream project's AI code policy (if any)
2. Ensure attribution trailers match the upstream project's requirements
3. Flag if the upstream project prohibits AI-generated contributions

### 7.3 Large Generated Blocks

When a single AI interaction produces more than approximately 100 lines of code that
will be committed with minimal editing:

1. Remind the developer to use the `Generated-by` trailer instead of `Assisted-by`
2. Recommend breaking the generated code into smaller, reviewable commits
3. Flag that extra review scrutiny is warranted

## 8. Enforcement

### 8.1 CI Checks

The CI pipeline (`.github/workflows/ci.yml`) should include:

- A commit trailer lint that verifies `Assisted-by` or `Generated-by` trailers are
  present when AI tools were used
- `cargo deny check licenses` to catch license compatibility issues
- `cargo clippy` with deny warnings to catch common AI-generated anti-patterns

### 8.2 Violations

Violations of this policy are handled through the standard code review process:

1. Reviewers flag the violation in PR comments
2. The author corrects the issue before merge
3. Repeated violations are escalated to the maintainer

This policy is not punitive. Its purpose is to maintain code quality, security, and
auditability in a security-critical project.
